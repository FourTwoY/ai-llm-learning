# Tool Error Cases

## 目标

今天补的是工具调用稳定性，重点是让 agent 在遇到异常时还能继续工作，而不是整个主循环直接崩掉。

## 模拟的 3 种失败

### 1. 工具参数为空

- 场景：`search_local_docs` 被调用时传入 `{"query": ""}`
- 风险：工具虽然被触发，但参数没有业务意义，容易报错或返回误导结果
- 处理：
  - 在进入工具前先做参数校验
  - 如果必填字段缺失或空字符串，返回可读错误
- 示例返回：

```text
工具 `search_local_docs` 执行失败。
错误原因：工具参数无效：`query` 不能为空字符串。
请根据这个错误调整参数，或改用别的方式回答用户。
```

### 2. 工具内部报错

- 场景：工具函数运行过程中抛出异常，比如网络波动、内部逻辑 bug、第三方接口异常
- 本次模拟：`broken_demo_tool(city="Shanghai")` 直接抛出 `RuntimeError`
- 处理：
  - 在主循环的工具执行阶段统一 `try/except`
  - 把 Python 异常转成用户和模型都能读懂的错误文本
- 示例返回：

```text
工具 `broken_demo_tool` 执行失败。
错误原因：工具内部异常：RuntimeError: 模拟内部异常：城市服务暂时不可用，city=Shanghai
请根据这个错误调整参数，或改用别的方式回答用户。
```

### 3. 工具返回空结果

- 场景：工具成功执行，但返回 `None`、空字符串、空列表、空字典
- 风险：如果主循环把它当作正常结果继续往下传，模型可能误以为已经查到了信息
- 本次模拟：`empty_demo_tool(keyword="RAG")` 返回空字符串
- 处理：
  - 执行后判断结果是否为空
  - 如果为空，统一返回“执行完成但没有命中”的提示
- 示例返回：

```text
工具 `empty_demo_tool` 执行完成，但返回了空结果。 请不要把它当成正常命中，建议换个关键词、补充上下文，或直接说明未找到信息。
```

## agent_loop_v2.py 做了什么

### 1. 参数解析保护

- 新增 `parse_tool_arguments()`
- 处理：
  - 参数为空时兜底为 `{}`
  - 非法 JSON 时返回可读错误
  - 非对象类型参数直接拦截

### 2. 参数校验

- 新增 `validate_tool_arguments()`
- 根据 `TOOLS` 里的 schema 检查必填字段
- 对空字符串做额外校验

### 3. 工具安全执行

- 新增 `execute_tool_safely()`
- 覆盖：
  - 未找到工具
  - 参数不匹配
  - 业务校验错误
  - 工具内部异常
  - 工具空结果

### 4. 主循环兜底

- `run_agent()` 里对以下环节都加了异常保护：
  - client 初始化
  - 模型请求
  - 参数解析
  - 单次工具执行
- 这样即使单个工具失败，也只会生成一条错误 tool message，不会直接让程序退出

## 当前效果

- 程序不会因为单次工具报错直接崩溃
- 模型能拿到结构化、可读的错误信息继续推理
- 空结果和真异常被区分开了，后续调试更清楚

## 可继续优化

- 给错误增加统一错误码，比如 `TOOL_PARAM_ERROR`、`TOOL_RUNTIME_ERROR`、`TOOL_EMPTY_RESULT`
- 把错误日志写入文件，便于排查真实线上问题
- 给不同工具做更细的字段级校验，而不只是通用校验
