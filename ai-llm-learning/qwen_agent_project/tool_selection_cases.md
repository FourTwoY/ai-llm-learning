# Tool Selection Cases

## 当前工具描述观察

- `get_current_time`：强调“只回答当前时间，不处理文档检索类问题”后，更容易和资料查询类请求区分开。
- `search_local_docs`：明确写出“本地资料、README、RAG 说明、chunk 相关内容”后，更容易把检索类问题路由到本地文档工具。
- 当前仍可能出现的误判：用户问题如果同时包含“现在”和“文档里现在怎么写”，模型可能只盯住时间词，或者被“查一下”误导去搜文档。

## 重点观察总结

### 1. 模型会不会乱调工具

- 会有一定概率乱调，尤其是用户问题同时出现“现在 / 当前”与“文档 / README / 配置”时。
- 如果工具描述太短，模型容易只抓住表面词，例如看到“现在”就去调时间工具。
- 对于泛知识问法，例如“RAG chunk 是什么”，模型也可能不调用任何工具，直接回答。

### 2. tool description 写清楚后，选择是否更准

- 会更准，尤其是给两个工具都写出“该用在什么场景”和“不要用在什么场景”之后，边界更清晰。
- `get_current_time` 加上“不要回答文档检索问题”这句后，能减少被“当前 / 现在”误导。
- `search_local_docs` 把“README、RAG 说明、chunk 相关内容”写进去后，能提高对本地检索类请求的命中率。

## 当前结论

- 最小双工具 demo 已经足够观察“工具选择”现象。
- 当前版本最值得观察的是：同一句里混合时间词和文档词时，模型是否还能稳定选对工具。
- 如果后续要继续优化，优先方向不是加更多代码，而是继续迭代 tool description 和 system prompt。
